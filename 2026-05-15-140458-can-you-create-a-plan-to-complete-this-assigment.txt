  Three architecture approaches

  A. Layered async pipeline with a Provider port (recommended)

  cli/         input loop, rendering, key bindings, spinner
  chat/        ChatSession (orchestrates turns), History, CancelToken
  providers/   base.py (Protocol) + anthropic.py, openai.py, gemini.py
  tools/       registry + weather.py + research.py + http client
  config/      env loading, provider selection

  One Provider Protocol with a single stream_turn(messages, tools, cancel) -> 
  AsyncIterator[Event] method that yields a normalized event stream (TextDelta,
  ToolUseStart, ToolUseArgsDelta, ToolUseEnd, TurnEnd). The chat loop is
  provider-agnostic. Tools are looked up by name from a registry — adding a tool
   is one file. Cancellation is a cooperative asyncio.Event passed into provider
   streams and tool calls; Ctrl+C sets it, current op unwinds, prompt returns.
  Second Ctrl+C at the prompt exits.

  Trade-off: Normalizing three SDKs' event streams costs ~50–80 lines per
  provider but means the chat loop is small and testable. Best fit for the
  "future extension" placeholder.

  B. Provider-thin, chat-loop-fat

  Each provider exposes the SDK's raw stream; the chat loop has three branches
  (one per provider) handling SDK-specific events. Smaller per-provider files,
  but the chat loop becomes a switch statement that's painful to extend.

  Trade-off: Lower upfront abstraction, higher long-term cost; works against the
   "pluggable providers" goal.

  C. Event-bus / actor style

  Components (LLM, tool runner, renderer, input) are independent tasks
  communicating via asyncio.Queue. Maximally decoupled, easiest to add
  observability/telemetry later. Overkill for ~250 lines of focused code — the
  queues become more LOC than the logic they coordinate.

  Trade-off: Elegant but speculative; the assignment rewards comprehensible
  code, not architectural flourish.

  Recommendation: A. It's the smallest design that genuinely supports three
  providers behind one chat loop, makes cancellation and tool dispatch
  unit-testable, and leaves clean seams for future tools/providers without dead
  code.

  Does A look right as the direction? If yes, I'll start presenting the detailed
   design (components → data flow → cancellation → error handling → file layout
  → testing) section by section for your approval.

⏺ Design — Section 1 of 6: Components

  Six modules, each with one job. Sizes are estimates so we know where to push
  back if something balloons.

  cli/app.py (~60 lines) — entrypoint. Parses args (--provider, --model,
  --resume <id>, --system <file>), loads config, constructs ChatSession, runs
  the input/output loop. Owns rich.Console and the prompt_toolkit PromptSession.
   Installs the SIGINT handler.

  cli/renderer.py (~50 lines) — turns the normalized event stream into terminal
  output: streams TextDeltas into a live region, shows a rich.Spinner while a
  tool call is pending, prints the tool name + truncated args/result above the
  spinner. No business logic.

  chat/session.py (~80 lines) — the orchestrator. Given a user message, calls
  Provider.stream_turn, fans events to the renderer, and when it sees a complete
   ToolUseEnd, invokes ToolRegistry.dispatch(name, args, cancel), appends the
  tool result to history, and loops back into the provider for the model's
  follow-up. Owns the CancelToken.

  chat/history.py (~40 lines) — append-only list of normalized turns ({role, 
  content, tool_calls, tool_results, ts}). Two operations: append(turn) and
  snapshot(). Persists each turn as one JSONL line to
  ~/.elyos_chat/sessions/<id>.jsonl. --resume last reads the newest file;
  --resume <id> reads a specific one.

  providers/base.py + providers/{anthropic,openai,gemini}.py (~50 + 80×3 lines)
  — Provider Protocol with stream_turn(messages, tools, cancel) -> 
  AsyncIterator[Event]. Each implementation translates our canonical message
  format to SDK calls and the SDK's stream events back to our canonical events
  (TextDelta, ToolUseStart, ToolUseArgsDelta, ToolUseEnd, TurnEnd, Error).
  Tool-schema translation also lives here (Anthropic/Gemini/OpenAI each shape
  tool JSON differently).

  tools/ (~30 + 40 + 40 lines) — registry.py (name → callable + JSON schema),
  http.py (single httpx.AsyncClient, base URL + X-API-Key header, timeouts,
  retry/backoff for transient failures), weather.py, research.py. Adding a tool
  = new file + one line in registry.py.

  config.py (~25 lines) — reads ELYOS_PROVIDER, ELYOS_MODEL, ELYOS_API_KEY,
  ANTHROPIC_API_KEY / OPENAI_API_KEY / GEMINI_API_KEY from env (+ optional .env
  via python-dotenv). Fails fast with a clear message if the selected provider's
   key is missing.

  Total estimate: ~600 lines (vs. the spec's "150–250"). The provider count is

  event normalization is the cost of paying for that flexibility. I'll flag this
   trade-off explicitly in the spec and in the Loom walkthrough.

⏺ Design — Section 2 of 6: Data flow for a turn

  A "turn" begins when the user presses Enter and ends when the assistant
  finishes its reply (after zero or more tool calls).

  User types text
        │
        ▼
  ChatSession.handle_user_input(text)
    • history.append(user_turn)
    • loop:
        events = provider.stream_turn(history.snapshot(), tools, cancel)
        pending_tool_calls = []
        async for ev in events:
            match ev:
                TextDelta(t)        → renderer.write(t)
                ToolUseStart(id,n)  → renderer.begin_tool(n)        # spinner on
                ToolUseArgsDelta(d) → accumulate JSON fragment
                ToolUseEnd(id,n,a)  → pending_tool_calls.append(...)
                TurnEnd(reason)     → break
                Error(e)            → raise / surface to renderer
        history.append(assistant_turn(text, pending_tool_calls))
        if not pending_tool_calls:
            return                                  # turn done
        results = await asyncio.gather(*[
            tools.dispatch(tc.name, tc.args, cancel) for tc in
  pending_tool_calls
        ])
        history.append(tool_results_turn(results))
        # loop back: feed results to provider for the follow-up

  Key choices

  - Provider abstraction is event-shaped, not response-shaped. Each provider
  yields the same five event types regardless of SDK. This is where the
  per-provider translation cost lives, and why the loop above is identical
  across Claude, Gemini, OpenAI.
  - Tool args are streamed. Anthropic and OpenAI both stream input_json_delta
  fragments; we buffer them per tool-use id and only dispatch on ToolUseEnd.
  Gemini delivers args atomically — the adapter still emits one ToolUseArgsDelta
   then ToolUseEnd for shape parity.
  - Parallel tool dispatch when the model asks for multiple. Models can return
  >1 tool use in a single turn (Anthropic especially). asyncio.gather runs them
  concurrently — important because research is 3–8s.
  - Tool-results turn always exists before re-entering the provider. No
  partial-state shortcuts; history is always coherent on disk, so resume works
  mid-conversation.
  - History snapshot is per-call. The provider gets a snapshot, not a mutable
  reference — keeps the provider stateless and testable.

  What gets persisted to JSONL per turn

  {"role":"user","content":"...","ts":...}
  {"role":"assistant","content":"...","tool_calls":[{"id":"...","name":"weather"
  ,"args":{...}}],"ts":...}
  {"role":"tool","results":[{"id":"...","content":"...","is_error":false}],"ts":
  ...}
  {"role":"assistant","content":"final answer","tool_calls":[],"ts":...}

  One JSON object per line; flush after each append. Resume rebuilds in-memory
  state by replaying the file.

  Does the flow look right? Specifically: are you happy with parallel tool 
  dispatch (vs. serial), and with persisting tool-call and tool-result turns as 
  separate JSONL records (vs. one combined record)? Both feel right to me but
  they're the two reversible-but-annoying-later decisions in this section.

⏺ Design — Section 3 of 6: Cancellation model

  The assignment calls cancellation out explicitly, so this section is
  deliberately specific.

  Primitives

  @dataclass
  class CancelToken:
      event: asyncio.Event = field(default_factory=asyncio.Event)
      def cancel(self): self.event.set()
      def cancelled(self) -> bool: return self.event.is_set()
      async def wait(self): await self.event.wait()

  One CancelToken per turn. It is threaded into provider.stream_turn(..., 
  cancel=token) and tools.dispatch(..., cancel=token). Both cooperate with it —
  they don't share threads, they share an awaitable signal.

  SIGINT handling

  The cli/app.py entrypoint installs a SIGINT handler via
  loop.add_signal_handler(signal.SIGINT, on_sigint). We do not rely on
  KeyboardInterrupt propagation, because it interacts badly with
  prompt_toolkit's own SIGINT semantics. The handler:

  ┌────────────────────────┬────────────────────────────────────────────────┐
  │   State when Ctrl+C    │                     Action                     │
  │        arrives         │                                                │
  ├────────────────────────┼────────────────────────────────────────────────┤
  │ Provider is streaming  │ cancel_token.cancel() — current turn unwinds,  │
  │ or tool call is        │ prompt returns. Renderer prints [cancelled].   │
  │ in-flight              │                                                │
  ├────────────────────────┼────────────────────────────────────────────────┤
  │ At the input prompt    │ Print "Press Ctrl+C again to exit." Arm a      │
  │ (no turn in flight)    │ 2-second exit window.                          │
  ├────────────────────────┼────────────────────────────────────────────────┤
  │ Second Ctrl+C within   │ Clean shutdown: close httpx.AsyncClient, flush │
  │ 2s at the prompt       │  JSONL, exit 0.                                │
  └────────────────────────┴────────────────────────────────────────────────┘

  How each layer cooperates

  - Provider stream: the SDK call is wrapped in asyncio.wait against
  cancel.wait(). On cancel, we close the SDK's stream (stream.close() /
  aclose()) and stop yielding events. We do not swallow the SDK's own
  cancellation exception — we translate it to a clean
  TurnEnd(reason="cancelled").
  - Tool dispatch: tools.dispatch runs the tool coroutine via asyncio.wait_for
  on a wrapper that races the tool against cancel.wait(). The httpx request is
  itself cancellable (async with client.stream(...) or await client.get(..., 
  timeout=...) inside an asyncio.Task); cancelling the task closes the
  underlying TCP connection.
  - Parallel tool gather: when multiple tools are in flight, cancelling one
  cancels all (asyncio.gather propagates cancellation to siblings; we set
  return_exceptions=True to keep the loop alive).
  - Mid-tool state: if a tool was cancelled, we still append a tool turn to
  history with {is_error: true, content: "cancelled by user"} so the
  conversation stays coherent — the model can see what happened on the next turn
   if the user types something new.

  What we explicitly do not do

  - We don't kill the process. Cancel means cancel a turn, not the app.
  - We don't retry a cancelled call. Cancellation is intent; retry is for
  transient failure.
  - We don't try to "reverse" partial streamed output already shown on screen.
  The cancelled assistant turn is recorded as truncated (tool_calls: [],
  content: "<whatever streamed so far>", cancelled: true).

  Why this is testable

  The CancelToken is the seam. A smoke test can: build a session with a fake
  provider that yields TextDelta events on a loop, call cancel_token.cancel()
  after the third event, and assert that the loop exits cleanly and history
  reflects truncation. No threading, no signals in the test path.

  Does this match what you want? In particular: do you agree with
  two-Ctrl+C-to-exit (a soft UX choice) and with recording cancelled turns in 
  history (so the model has context on a follow-up)?

⏺ Design — Section 4 of 6: Error handling & API quirk strategy

  This is the most heavily-weighted dimension of the assignment, so I'll be
  specific about the framework — the actual quirks we'll discover via the probe
  script, not predict here.

  Two layers of error handling

                  ┌─────────────────────────────────────────┐
                  │  Layer 1: tools/http.py (transport)     │
                  │  • timeouts, retries, network errors    │
                  │  • JSON validity, HTTP status           │
                  │  • returns Result[T] — never raises out │
                  └─────────────────────────────────────────┘
                                    │
                  ┌─────────────────▼───────────────────────┐
                  │  Layer 2: tools/weather.py, research.py │
                  │  • response-shape validation            │
                  │  • known-quirk normalisation            │
                  │  • returns dict-shaped tool result      │
                  └─────────────────────────────────────────┘

  Tools always return something the model can read. A failure becomes {"error": 
  "...", "guidance": "..."} so the model can decide to retry, ask the user, or
  apologize — it does not become a Python exception that kills the turn.

  tools/http.py — the chokepoint

  class ToolHttpClient:
      base_url: str
      api_key: str
      client: httpx.AsyncClient   # timeout=httpx.Timeout(connect=5, read=15)

      async def get(self, path, params, cancel) -> Result:
          # 1. retry transient: 429, 502, 503, 504, ReadTimeout, ConnectError
          # 2. backoff: 250ms · 2^n + jitter, max 3 attempts
          # 3. respect Retry-After if present
          # 4. abort early if cancel.cancelled()
          # 5. content-type guard: if not JSON, return 
  Result.err("non-json:<excerpt>")
          # 6. status guard: 4xx → no retry (caller's fault), surfaced as error

  The retry policy is deliberately small: research is already 3–8s, so we cap
  attempts at 3 and total budget at ~30s before we hand a clear timeout error
  back to the model.

  The quirks pipeline

  scripts/probe_api.py        → docs/api-findings.md       →
  tools/{weather,research}.py
     (systematic probes)         (one row per quirk with      (handlers
  reference findings
                                  repro, hypothesis, impact)   by name in
  comments)

  scripts/probe_api.py runs targeted experiments and prints a structured report.
   Categories it probes:

  - Auth: missing key, wrong key, key in wrong header case (x-api-key vs
  X-API-Key).
  - Param surface: missing required param, empty string, very long string,
  unicode, URL-encoded vs raw spaces, leading/trailing whitespace, repeated
  query params.
  - Determinism: 3× same location, 3× same topic — same body? Latency variance?
  - Timing: measure p50/p95/p99 for both endpoints; check whether research ever
  returns <1s (cached?) or >10s (degraded?).
  - Response shape: are fields stable across calls? Any nulls? Any
  prose-in-fields? Is the JSON valid every time?
  - Status-code semantics: does the API use 200 for "success" only, or also for
  soft errors with error in body?
  - Concurrency: 10 parallel research calls — rate limit? Coalesced?
  Out-of-order completion?

  Each finding lands in docs/api-findings.md like:

  ### F-03: research endpoint returns 200 with {"error": "..."} for empty topic
  - Repro: curl ...?topic=
  - Impact: must treat 200 as success only when expected schema present
  - Handler: tools/research.py — schema_guard()

  async def research(args: dict, cancel: CancelToken, http: ToolHttpClient) ->
  dict:
      topic = (args.get("topic") or "").strip()
      if not topic:
          return {"error": "missing topic", "guidance": "ask the user for a 
  research topic"}
      result = await http.get("/research", {"topic": topic}, cancel)
      if result.is_err:
          return {"error": result.error, "transient": result.is_transient}
      body = result.value
      # quirk handlers go here, each referencing a finding ID in a one-liner 
  comment
      return _normalise(body)

  LLM-side errors (rate limits, malformed tool args from the model) are caught
  in the provider adapter and surfaced as Event.Error, which the chat loop
  renders cleanly and offers to retry.

  What's deliberately not here

  - No fancy circuit breaker — three retries with jittered backoff is enough for
   a 1-week assignment.
  - No persistent retry queue — if a turn fails terminally, the user sees it and
   decides what to do.
  - No automatic schema-drift detection — if the API changes shape after we
  ship, the handlers fail loudly with a clear message rather than silently
  degrading.

  Two asyncio.Tasks run concurrently, each fed by asyncio.Queues, joined by a
  single CancelToken:

  1. client_pump — async for msg in ws.iter_*(). If binary, push to
  upstream_audio_queue. If JSON cancel → trigger Realtime's response.cancel
  event. If JSON stop → tear down.
  2. upstream_pump — async for event in realtime_ws. Branch on event.type:
    - input_audio_buffer.speech_started → {type: "speech_started"} to client
    - response.audio.delta → forward binary to client (assistant voice)
    - response.audio_transcript.delta → {type: "transcript_assistant_delta", 
  text: ...} + append to in-memory assistant buffer for history
    - conversation.item.input_audio_transcription.completed → {type: 
  "transcript_user_done", text: ...} + append user turn to JSONL
    - response.function_call_arguments.done → tool call complete. Dispatch via
  ToolRegistry, emit tool_use_end + tool_result to client, then send
  conversation.item.create with function_call_output upstream to feed result
  back to the model.
    - response.done → {type: "turn_end", reason: "stop"} + append assistant turn
   (with accumulated transcript) to JSONL.
    - error → {type: "error", message: ...}.

  Audio format negotiation — Realtime expects PCM16 mono at 24 kHz. The
  browser's MediaRecorder outputs audio/webm;codecs=opus by default — wrong
  format. So web-ui/src/lib/voice.ts uses an AudioWorkletNode instead:
  - Capture raw mic via getUserMedia → AudioContext (sample rate negotiated to
  24 kHz).
  - An AudioWorkletProcessor slices the stream into ~20 ms frames, converts
  Float32 → Int16 little-endian, and postMessages the buffer to the main thread.
  - Main thread sends the ArrayBuffer over the WebSocket as binary.

  For playback: server sends Int16 LE binary chunks → client converts to Float32
   → enqueues into an AudioBufferSourceNode chain (FIFO playback). A single
  playback_queue ensures chunks play in order even when they arrive out of
  order.

  "name": ev.name, "args": args})

  type Session = { id: string; title: string; updatedAt: number; messageCount:
  number };

  interface Store {
    // Settings
    provider: "anthropic" | "openai" | "gemini";
    model: string | null;
    ttsEnabled: boolean;              // narrate assistant replies (text mode 
  only)
    voiceName: "alloy" | "verse" | "shimmer";  // voice mode voice

    // Connection
    textWs: WebSocket | null;
    voiceWs: WebSocket | null;
    mode: Mode;

    // Data
    sessionId: string | null;
    messages: Message[];
    sessions: Session[];              // sidebar list
    pendingTools: Record<string, { name: string; startedAt: number }>;  // 
  tool_use_id -> meta (drives spinners)

    // Actions
    connectText: (sessionId: string | "new" | "last") => void;
    sendUserMessage: (text: string) => void;
    cancelCurrentTurn: () => void;
    switchSession: (id: string) => void;
    startVoiceMode: () => Promise<void>;
    stopVoiceMode: () => void;
    setProvider: (p: string) => void;
    setModel: (m: string | null) => void;
    toggleTts: () => void;
    loadSessions: () => Promise<void>;

    // Internal — called by WS handlers
    _handleTextEvent: (ev: ServerEvent) => void;
    _handleVoiceEvent: (ev: ServerEvent) => void;
  }

  Why zustand: small, no provider boilerplate, plays well with WebSocket
  callbacks (handlers call store actions directly). React Query / Redux Toolkit
  would be heavier than needed for a single-screen app.

  Component contracts (concise)

  - <TopBar> — purely controlled by store. Provider dropdown calls setProvider
  which sends an update_settings message over the text WS. Voice toggle calls
  startVoiceMode / stopVoiceMode (these manage voiceWs lifecycle).
  - <Sidebar> — calls loadSessions() on mount, polls every 30s to pick up new
  sessions written by the CLI app. Clicking an item calls switchSession(id)
  which closes any open text WS and reconnects with hello { session_id: id }.
  - <MessageList> — virtualized via shadcn's <ScrollArea> + react-window only if
   message count > 200. For most sessions, plain map. Auto-scrolls to bottom on
  every store update (with a "you've scrolled up" detector that pauses
  auto-scroll).
  - <AssistantBubble> — when streaming: true, shows a blinking cursor ▍ after
  the text. Renders markdown (assistant outputs are often markdown-ish; use
  react-markdown + remark-gfm).
  - <ToolRow> — when pendingTools[id] exists, shows a shimmering card with the
  tool name and a spinner. When tool_result lands, replaces with a static card
  showing the tool name, args (collapsed), and a one-line result preview (click
  to expand the full JSON).
  - <VoiceBar> — visible only in voice mode. Uses
  AnalyserNode.getByteFrequencyData to render a 32-bar mic level meter. Listens
  for speech_started/speech_stopped to color the meter. A red "stop" button
  calls stopVoiceMode. A pause button sends cancel to interrupt the assistant
  mid-sentence.
  - <Composer> — Multi-line textarea (Enter submits, Shift+Enter newlines). The
  mic button hold-to-talk in text mode (sends one user message after release);
  in voice mode the mic button doesn't exist (continuous capture). Disabled when
   mode === "text-streaming".

  Streaming flow on the frontend (text mode example)

  User types "weather in London", presses Enter
    ↓ Composer.onSubmit
  store.sendUserMessage(text)
    ↓ optimistically appends user Message; sets mode="text-streaming"
    ↓ textWs.send({type:"user", content:text})
                                                   ← server: user_echo (already
  optimistic, no-op)
                                                   ← server: tool_use_start
  {id:"t1", name:"weather"}
    ↓ pendingTools["t1"] = {name:"weather", startedAt:Date.now()}
                                                   ← server: tool_use_end
  {id:"t1", args:{...}}
                                                   ← server: tool_result
  {id:"t1", content:{...}}
    ↓ remove from pendingTools; append a ToolRow message
                                                   ← server: text_delta
  {text:"It's"}
                                                   ← server: text_delta {text:"
  12°C"} ...
    ↓ append/extend last assistant message; streaming: true
                                                   ← server: turn_end
  {reason:"stop"}
    ↓ streaming: false; mode: "idle"; Composer re-enabled

  TTS in text mode

  When ttsEnabled is on AND mode === "text-streaming", the store accumulates
  text deltas into sentence-sized chunks (split on ., ?, !, \n) and feeds each
  chunk to window.speechSynthesis.speak(new SpeechSynthesisUtterance(chunk)).
  Free, zero-deps, works offline. The voice mode TTS is separate (Realtime
  serves the audio directly).

  Does the component+store shape look right? In particular:
  - zustand as the only state lib (no React Query) — small surface for an
  assignment.
  - Browser-native SpeechSynthesis for text-mode TTS (vs. an API-based TTS) —
  free + instant.
  - Sessions list polls every 30s to discover sessions created by the CLI — a
  small touch but lets the web UI seamlessly continue conversations started from
   the terminal.

⏺ Design — Section 6 of 7: Error handling & edge cases

  The CLI's "tools always return something the model can read" rule still
  applies — the registry layer is unchanged. New failure surfaces are the
  transport (WebSocket) and voice pipeline.

  WebSocket failure modes

  Failure: Connection drops mid-turn
  Detection: ws.onclose + reason code
  Behavior: Show toast "connection lost — reconnecting"; auto-reconnect with
    exponential backoff (250 ms · 2^n, max 5 attempts, capped at 10 s). On
    reconnect, replay hello with the current session_id. The assistant turn that

    was streaming gets marked cancelled: true in UI; server-side history already

    wrote whatever streamed.
  ────────────────────────────────────────
  Failure: Server is down on first load
  Detection: First onopen never fires; onerror fires fast
  Behavior: Banner at top: "Server not reachable. Is python -m web.server
    running?". Retry button.
  ────────────────────────────────────────
  Failure: Server pushes {type: "error", transient: true}
  Detection: Type guard on incoming events
  Behavior: Show toast; keep WS open. Don't reset state.
  ────────────────────────────────────────
  Failure: Server pushes {type: "error", transient: false}
  Detection: Same
  Behavior: Show toast; reset mode to idle; re-enable composer.
  ────────────────────────────────────────
  Failure: Backpressure on the server (LLM provider slow)
  Detection: Token gap > 30 s
  Behavior: Show a subtle "still thinking…" hint after the cursor. No automatic
    action.

  Reconnect logic has one subtlety: don't replay the in-flight user message
  after reconnect — the server-side history already recorded the user turn, and
  the assistant turn was marked cancelled with whatever text streamed. User can
  retry by hitting up-arrow → Enter (composer keeps last sent message).

  Voice-mode failures

  Failure: Mic permission denied
  Detection: getUserMedia throws NotAllowedError
  Behavior: Toast with a "open browser settings" link; voice toggle reverts to
    off.
  ────────────────────────────────────────
  Failure: Mic device disappears mid-session (e.g. unplug)
  Detection: MediaStreamTrack.onended
  Behavior: Stop voice mode gracefully; show toast "microphone disconnected".
  ────────────────────────────────────────
  Failure: Realtime upstream WS drops
  Detection: Upstream onclose non-1000
  Behavior: Send {type: "error", message: "voice service interrupted"} to
  client;
    client shows toast + stops playback queue + reverts to idle. No
    auto-reconnect for voice (interrupting an active voice conversation with a
    reconnect is worse than just stopping).
  ────────────────────────────────────────
  Failure: Audio playback underrun (queue empty)
  Detection: Detect in AudioWorklet
  Behavior: Insert 20 ms of silence (avoids clicks); emit {warning: "audio 
    underrun"} if it happens >3× in a row.
  ────────────────────────────────────────
  Failure: Tool call in voice mode fails
  Detection: Same as text mode — registry returns {error: ...}
  Behavior: The model speaks an apology naturally. UI shows ✗ tool card.
    Critical: don't suppress the upstream function_call_output — Realtime needs
  a
     response even on tool failure, else it hangs.
  ────────────────────────────────────────
  Failure: User speaks while assistant speaks ("barge in")
  Detection: Realtime's input_audio_buffer.speech_started event arrives while
    playback queue has items
  Behavior: Auto-clear playback queue + send response.cancel upstream.
  Realtime's
    server-side VAD handles the actual turn-taking — we just stop playback so
  the
     user doesn't hear the cut-off assistant phrase.
  ────────────────────────────────────────
  Failure: Browser tab backgrounded during voice mode
  Detection: document.visibilityState === "hidden"
  Behavior: Keep the WS open (Chrome throttles timers in background tabs but
    WebSocket messages still arrive). Playback continues. Don't auto-pause —
  user
     expectation in voice chat is "it keeps going while I look away briefly".

  Provider errors flowing through both modes

  The existing chat/session.py already catches provider exceptions and surfaces
  them as Error events. The web layer translates those into {type: "error"}
  JSON. Three categories the UI distinguishes:

  1. Rate limit (provider returns 429): toast "Rate limited — wait a moment".
  Don't auto-retry; user retries.
  2. Auth (401/403): banner "API key invalid for <provider>. Check your .env."
  Modal helps locate the file.
  3. Other: generic toast with the provider's error message.

  History coherence across modes

  Voice and text turns land in the same JSONL file. Two non-obvious edges:

  - Resume into voice from a text-only session: history loads fine. Realtime's
  conversation.item.create accepts text user/assistant turns as context — we
  send the last ~20 turns when the voice session opens.
  - User switches mid-turn: blocked. The voice toggle is disabled while mode ===
   "text-streaming" and vice versa. Avoids ambiguous state.

  CSRF / origin

  FastAPI checks Origin header on WS upgrade (localhost:5173 and localhost:8000
  only in dev). In production, the SPA is served by the same FastAPI app so
  Origin matches. No tokens needed since this is single-user local.

  Cancellation across the wire

  - Text mode: client sends {type: "cancel"} → server calls
  session.cancel_current() (existing CLI primitive). Same CancelToken plumbing
  as the terminal app — no new code path.
  - Voice mode: client sends {type: "cancel"} → server cancels the shared
  CancelToken (interrupts any in-flight tool dispatch) AND sends {type: 
  "response.cancel"} upstream to Realtime (stops generation + audio).

  One Esc keypress in the UI triggers cancel for whichever mode is active.
  Composer also has a visible stop button. Two presses within 1 s in voice mode
  exits voice mode entirely (mirrors the CLI's "two Ctrl+C exits" pattern).

  Server lifespan

  FastAPI lifespan:
  - Startup: build one shared ToolRegistry + ToolHttpClient (no per-connection
  httpx clients — the registry is request-safe). Validate provider keys early
  (warn on missing, don't fail — user might switch providers in UI).
  - Shutdown: await http.aclose(), cancel all in-flight WS handlers gracefully.

  What's deliberately NOT handled

  - No multi-tab session locking. If you open two tabs to the same session, both
   write to the JSONL — second write wins on resume. UI shows a tiny warning
  "another tab is on this session" via BroadcastChannel, but the user is
  responsible for not racing themselves.
  - No offline mode / service worker. Tab must be online.
  - No quota / rate limiting on the FastAPI server. Single-user local; YAGNI.

  One non-obvious call worth flagging:
  - Voice mode does not auto-reconnect on drop (text mode does). Reconnecting in
   the middle of a live conversation is jarring; stopping cleanly + letting the
  user re-enable voice is better UX.

  Last section (testing) coming up. Anything to adjust here first — especially
  the voice no-auto-reconnect choice, or the same-JSONL-across-modes decision?
